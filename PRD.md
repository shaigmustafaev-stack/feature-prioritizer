# ProductHub — PRD

## 1. Проблема

Продакт-менеджеры работают с множеством разных инструментов: идеи в документах, приоритизация в таблицах, roadmap отдельно, аналитика в BI-системах, фидбек в чатах и тикетах.

В результате:
- Нет одного места, где видно состояние продукта
- Фичи, метрики и фидбек не связаны между собой
- Много ручной работы
- Сложно быстро принять решение о приоритетах

ProductHub — единый workspace для продактов, который объединяет ключевые инструменты и усиливает их AI.

Первый этап: внутренние продуктовые команды Битрикс24.
- Product Manager / Senior PM / Product Owner

Масштабирование: любые продуктовые команды.

## 3. Инструменты

### 3.1 Приоритизатор фич — (готов)
Оценка и сортировка фич по RICE/ICE.
**AI-усиление:** Claude предлагает оценки по описанию фичи.

### 3.2 Генератор фич и описаний
Превращает сырую идею в структурированное описание.
**Что делает:**
- Принимает идею в свободной форме
- Задаёт уточняющие вопросы (через AI)
- Генерирует: описание фичи, User Stories, Acceptance Criteria
- Можно отправить фичу в приоритизатор

**AI-усиление:** Ядро инструмента — Claude генерирует документацию.

### 3.3 Roadmap продукта
Визуальная канбан-доска с этапами.
**Этапы:** Идея → Исследование → Дизайн → Разработка → Тестирование → Прод
**Что делает:**
- Карточки фич с drag-and-drop между колонками
- Связь с приоритизатором (фича переносится одним кликом)
- Фильтры по приоритету, статусу, ответственному

**AI-усиление:** Claude предлагает в какую колонку поставить фичу на основе готовности.

### 3.4 Аналитика продукта — (готов)
Ввод метрик → дашборд с графиками + AI-выводы.
**Что делает:**
- CRUD дашбордов с JSONB-хранением в Supabase
- Ввод метрик с сегментацией (по платформам, тарифам и т.д.) и периодами (месяц + год)
- Автоматический выбор типа графика (line/bar/pie/horizontal-bar) по правилам
- KPI-карточки с дельтой (▲/▼)
- Шаринг: публичный read-only URL по share_id
- Auto-save с дебаунсом 5 сек

**AI-усиление:** OpenRouter API (free модель) анализирует метрики — факт, гипотеза, рекомендация под каждым графиком. Rate limit: 1 запрос / 30 сек.

### 3.5 Сбор фидбека и багов
Единая точка входа для пользовательского фидбека.
**Что делает:**
- Список жалоб, пожеланий, багов
- Категоризация и теги
- Связь с фичами (фидбек → фича в приоритизаторе)
- Анализ повторяющихся проблем

**AI-усиление:** Claude автоматически категоризирует фидбек и находит паттерны.

## 4. Связи между инструментами

Главная ценность — инструменты связаны:

```
Фидбек → "Создать фичу" → Приоритизатор → "В roadmap" → Roadmap
                                ↓
                          Генератор фич
                          (описание + User Stories)
                                ↓
                           Аналитика
                    (данные для обоснования)
```

Фича — центральная сущность. Она рождается из фидбека или идеи, получает описание в генераторе, оценку в приоритизаторе, и движется по roadmap до прода.

## 6. User Flow

### Основной путь
1. Продакт заходит — видит главную с инструментами
2. Выбирает инструмент по задаче
3. Работает внутри инструмента
4. Результат связывается с другими инструментами

### Сценарии
**Планирование спринта:** Приоритизатор → сортирует фичи → отправляет топ в Roadmap

**Новая идея:** Генератор → описывает фичу → отправляет в Приоритизатор → оценивает

**Обработка фидбека:** Фидбек → анализирует жалобы → создаёт фичу → в Приоритизатор

**Обзор продукта:** Аналитика → смотрит метрики → принимает решения

## 8. Модель данных (высокоуровневая)

### Основные сущности
**User** — пользователь (id, email, name, role)
**Project** — проект/продукт (id, name, owner_id)
**Feature** — фича, центральная сущность (id, name, desc, rice_scores, status, project_id, created_by)
**Feedback** — единица фидбека (id, text, category, feature_id, project_id)
**RoadmapColumn** — колонка доски (id, name, order, project_id)

### Связи
- User → много Projects
- Project → много Features
- Project → много Feedbacks
- Feature → один RoadmapColumn
- Feedback → может быть связан с Feature
