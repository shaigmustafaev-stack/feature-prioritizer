# Commit Guidelines

Короткие правила, чтобы не путаться в истории и легко откатываться.

## 1) Один коммит = одна цель
- Хорошо: `fix(rice-mobile): починить tap по InfoTip`
- Плохо: `fix: mobile + supabase + тесты + docs`

## 2) Формат сообщения
Используем шаблон:

`type(scope): кратко что сделано`

Типы:
- `feat` — новая функциональность
- `fix` — исправление бага
- `test` — добавление/обновление тестов
- `docs` — документация
- `refactor` — рефакторинг без изменения поведения
- `chore` — инфраструктурные/служебные изменения

Примеры:
- `feat(supabase): хранить фичи в облаке`
- `fix(rice-mobile): исправить порядок полей на мобиле`
- `test(e2e): добавить mobile UX smoke-тест`
- `docs(process): описать правила коммитов`

> Тип — английский (`feat`, `fix` и т.д.), описание — **русский**.

## 3) Что обязательно перед push в main
- `npm test`
- `npm run build`
- UI-задачи: `npm run test:e2e` (или целевой e2e-спек)

## 4) Что не делать
- Не смешивать UI, backend и docs в одном коммите.
- Не пушить без проверки сборки и тестов.
- Не менять историю `main` через force-push без отдельного согласования.

## 5) Точки отката (checkpoints)
Перед крупной задачей создаём tag:

`git tag -a checkpoint-YYYY-MM-DD-<short-name> -m "checkpoint before <task>"`

Пример:
- `checkpoint-2026-03-10-before-supabase-sync`
